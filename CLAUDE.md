# CLAUDE.md

## プロジェクト概要

東京の地下鉄・JR・私鉄を対象にした障がい者割引運賃の最短経路探索アプリ。
単一のHTMLファイル（index.html）+ version.js + sw.js で構成。

## アーキテクチャ

### UI
- SPA構成: #search / #results / #lines / #about のハッシュルーティング + ボトムナビ
- PWA: version.js でバージョン一元管理。sw.js が importScripts で参照し、CACHE_NAME を自動連動
- 結果表示: 初期30件、「もっと見る」で全件。詳細は `<details>` 展開時に遅延レンダリング

### グラフ構造
- 各駅がノード（671駅）、隣接駅間が辺（重み＝営業キロ）
- 同名駅は自動接続(0km)、異名乗換は `CROSS_TRANSFERS` で定義(0km)
- `lineFamily()`: JR在来線全12路線を同一系統 "JR" として統合（改札内乗換扱い）
- 駅コンプレックス: Union-Find で同一駅(grp)を管理。乗換判定・重複排除に使用

### 運賃計算
- 経路を会社(operator)ごとのセクションに分割し、各社の運賃テーブルで個別計算して合算
- JR特定区間: `JR_SPECIAL` に登録された駅ペアは距離ベース運賃より優先
- JR 100km超: 5割引（障害者割引）

## 経路探索アルゴリズム

### 概要

Dijkstra法をベースに、パラメータ変更・辺ブロック・駅ブロック・経由点強制の組み合わせで
多様な経路候補を生成する。**750ms〜2000msの時間予算内**で最大100件(dedup後)を目標に探索。

商用乗換案内サービス（NAVITIME、Yahoo!乗換案内など）はYenのK最短経路アルゴリズムや
RAPTOR（時刻表ベースの到着時刻最適化）を使用している。本アプリは時刻表を持たないため、
Dijkstra法＋多戦略ブロッキングで候補を多様化するアプローチを取っている。

### Dijkstra変形: dijkstraP(srcSet, dstSet, penalty, transferCost, ccMul)

| パラメータ | 説明 |
|---|---|
| srcSet / dstSet | 出発/到着駅IDのSet（同名・異名乗換を含む） |
| penalty | 辺キー→ペナルティのMap（ブロック対象の辺に999や99999を設定） |
| transferCost (tc) | 乗換時に加算する探索ペナルティ（高いほど乗換を避ける） |
| ccMul | 会社またぎコスト倍率（0=会社ペナルティなし、1=通常） |

乗換辺（w=0）を通過するとき:
- 異ファミリー間: `tc + companyCost(opA, opB) * ccMul`
- 同ファミリー異路線: `INTRA_FAMILY_COST (0.1km)`

### 探索フェーズ

```
Phase 1a: 基本探索
  TC_ALL = [0, 0.3, 0.6, 1, 1.5, 2.5, 4, 7, 10, 15, 20, 30, 50]
  各tcでDijkstra実行 × ccMul=1とccMul=0の両方

Phase 1b: 経由点探索（via-point search）
  全駅コンプレックス内の異会社ペアを列挙（同名・異名問わず）
  各ペア(mid, next)で:
    leg1 = dijkstra(src → mid, tc=50, ccMul=0)  ← 同一路線到達を優先
    leg2 = dijkstra(next → dst, tc=50, ccMul=0)
    合成して候補に追加

Phase 2: 反復ブロッキング（新ルート発見のたびに再帰的に展開）
  見つかった各ルートに対して:
  - 辺ブロック: 各辺を999ペナルティでブロック × TC_ALL × ccMul=0/1
  - 駅ブロック: 各中間駅(grp)の全辺を99999でブロック × TC_ALL × ccMul=0/1
  - 2辺同時ブロック: 辺ペアを同時ブロック × TC_FAST × ccMul=0/1

Phase 3: 残余辺ブロック
  sig重複で弾かれたルートの辺も収集済み(allEdges)
  未試行の辺をブロックして追加探索

Phase 4: 網羅的駅ブロック
  全駅グループをハブ順(接続数降順)にブロックして探索

Phase 5: ルート回廊ブロック
  見つかった各ルートの全中間駅を一括ブロックして完全別経路を強制探索
```

### タイムアウト制御

| 条件 | 動作 |
|---|---|
| 750ms経過 & dedup後50件以上 | 終了 |
| 1500ms経過 | 50件未満でも終了 |
| 2000ms経過 | 無条件で強制終了 |
| dedup後100件以上 | 時間に関係なく終了 |

### 重複排除

1. **routeSig** (探索中): 個別路線コード × 始終駅grpで署名。同一sigでも乗換が少ない or 距離が短いルートで上書き
2. **routeShape** (最終): lineFamily × 始終駅grpで署名。同一(shape, 距離, 乗換数)内は駅数最少を採用
3. **乗換回数フィルタ**: 各(shape, 距離)グループで乗換回数は最少〜最少+1回に制限

### パス後処理
- 先頭・末尾の同一grpノードを除去（出発/到着駅の正規化）
- **中間の同一grpチェーン除去**: A(C)→B(I)→C(M)が全て同一grpなら→A(C)→C(M)に短縮

### おすすめスコア（「おすすめ」ソートの基準）

`score = distance + transfers × 3 + paidCross × 3 + toeiCross × 0.3`

スコアが小さいほど上位。各項の意味（単位はkm換算）:

| 項 | 内容 | 重み |
|---|---|---|
| `distance` | 営業キロ(km) | 1.0（基準） |
| `transfers` | 乗換回数 | 1回 = 3km相当 |
| `paidCross` | 有料会社またぎ回数（メトロ↔JR等） | 1回 = 3km相当 |
| `toeiCross` | 都営が絡むまたぎ回数 | 1回 = 0.3km相当（都営は無料なのでほぼ無視） |

つまり「距離をベースに、乗換1回と有料会社またぎ1回をそれぞれ3km分のペナルティとして加算した総合評価」。運賃そのものは含まないため、最安を見たい場合は「運賃」ソートを使う。

### 商用サービスとの違い

| 観点 | 本アプリ | 商用サービス |
|---|---|---|
| 探索基準 | 営業キロ（距離最短） | 時刻表ベース（到着時刻最早） |
| アルゴリズム | Dijkstra + ブロッキング + via-point | Yen's K-shortest / RAPTOR / CSA |
| 候補多様化 | パラメータ変更 + 辺/駅ブロック | K最短経路 or 制約付き再探索 |
| 時刻表 | なし | あり（リアルタイム） |
| 運賃 | 距離ベース計算 | 運賃データベース参照 |
| 所要時間 | 表示なし | 主要な評価軸 |

## v2.html（実験版: RAPTOR風 + Plateau + 最安運賃探索）

`v2.html` は index.html(v1) と同じデータ・UIで、経路探索アルゴリズムだけを置き換えた実験版。
**主目的は最安運賃の発見**（距離は重視しない）。SW登録なし。

### アルゴリズム構成（4段ハイブリッド、1500ms物量）

1. **RAPTOR風ラウンド探索** (`roundShortest`): 状態(駅, 乗換回数k)のDijkstra。
   乗換回数別の最短距離を求め、「乗換を増やすと距離が縮む」Pareto最適のみ採用
2. **Plateau法** (`plateauRoutes`): 出発側・到着側の最短経路木で共有される連続区間(plateau)から
   構造的に異なる代替経路を生成。Yen'sと違い同距離微変形が出ない
3. **運賃近似Dijkstra 固定バイアス** (`fareApproxDijkstra`): 辺重み = 都営≈0・有料=実距離。
   各社を優先した4パターン(metro/jr/tx/keisei優先)で探索
4. **運賃近似 + 会社重みランダム化 物量**: 会社の魅力度をランダムに変えて1500ms回し、
   全候補を運賃計算して最安を拾う

### 最安運賃の核心（なぜ会社重みランダム化が必要か）

最安運賃は距離最短とは別物。会社ごとに初乗り額・運賃曲線が違うため、
「有料距離最小」≠「運賃最小」。例:
- メトロ初乗り180円 > JR短距離(3駅)160円 → 距離だけ見るとメトロ1駅を選んで割高になる
- そのため「どの会社を優先するか」を毎回ランダムに変えて物量で回し、安い組み合わせを引く

### v1が見逃しv2が見つけた最安ルートの例

| 区間 | v1最安 | v2最安 | v2が安い理由 |
|---|---|---|---|
| 北千住→吉祥寺 | 390円 | 350円 | 起点をメトロ(180円)でなくTX(140円・半額)にする |
| 池袋→品川 | 350円 | 320円 | JR直通(350円)を都営無料区間で2分割→JR160円×2 |
| 東京→新宿 | 160円 | 160円 | JR短区間(東京→馬喰町160円)+都営新宿線無料 |

共通パターン = **有料区間を短く切り、TX(半額)や都営(無料)の安い区間でつなぐ**。
ただし遠回り・乗換増になりがちで、実用性では距離最短ルートに劣る場合もある。

### v1 vs v2 性能比較

| 観点 | v1 (ブロッキング+via-point) | v2 (RAPTOR+Plateau+運賃物量) |
|---|---|---|
| 速度 | 300〜2000ms | 1500ms固定（物量）|
| 候補数 | 150〜220件 | 40〜170件 |
| コード | 5フェーズ+補助多数 | 4アルゴリズム |
| 最安発見 | 距離スコア優先で見逃しあり | 同等以上（会社分割も発見）|
| デフォルトソート | おすすめ | 運賃 |

### 商用乗換案内が使う他の工夫（参考）

- **RAPTOR/CSA**: 時刻表ベースで到着時刻最早を最適化（要GTFS）
- **Transfer Patterns** (Google): 経路パターンを事前計算しクエリ時はDB参照で爆速
- **Contraction Hierarchies**: 重要でないノードを収縮して階層化（道路ナビ主流）
- **Plateau法 / Penalty法**: 代替経路生成（v2はPlateauを採用）
- **乗換時間モデル**: 駅・路線ペアごとの実測徒歩時間
- 本アプリは時刻表を持たないため、距離+運賃近似+物量で代替
